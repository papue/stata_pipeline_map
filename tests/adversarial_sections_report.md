# Adversarial Section-Header Extractor Report

Tested against `src/data_pipeline_flow/parser/section_extract.py`.

---

## False Positives Found and Fixed

### FP-01 — IPython cell magics detected as section headers (Python)

| Field | Value |
|-------|-------|
| Input | `# %%time`, `# %%timeit`, `# %%capture` |
| Expected | `[]` (IPython magic, not a section) |
| Actual (before fix) | `[Section(1, 1, "time")]`, etc. |
| Root cause | Cell marker pattern `r"^\s*#\s*%%\s*(.*)$"` used `\s*` (zero-or-more spaces) after `%%`, so `%%time` matched with group(1) = `"time"`. IPython magics have no space between `%%` and the magic name. |
| Verdict | **Fixed** |
| Fix | Changed cell marker pattern to `r"^\s*#\s*%%(\s+(.+?))?\s*$"` — requires whitespace after `%%` or bare `%%`. IPython magics (`%%time`) have no space and no longer match. Cell handler updated to use `m.group(2)`. |

---

### FP-02 — R console output pasted as comments detected as sections (Python and R)

| Field | Value |
|-------|-------|
| Input (Python) | `## [1] 0.234` |
| Input (R) | `## [1] TRUE`, `## [1] "some string"` |
| Expected | `[]` (R console output, not a section) |
| Actual (before fix) | `[Section(1, 1, "[1] 0.234")]`, etc. |
| Root cause | The multi-hash patterns (`## Title`) matched any non-empty text after `##`. R console output like `[1] 0.234` passes all title-validity checks. |
| Verdict | **Fixed** |
| Fix | Changed multi-hash patterns for both Python and R to `r"^\s*#{2,}(?:\s+)(?!\[\d)(.+?)..."` — the `(?:\s+)` requires at least one space after `##` (valid section headers always have one), and `(?!\[\d)` is a negative lookahead that rejects titles starting with `[<digit>` (the R console output prefix). |

## False Negatives Found and Fixed

### FN-01 — Stata decorated-numbered headers missed

| Field | Value |
|-------|-------|
| Input | `* -- 1. Preparation --`, `* === 1. Section ===`, `* --- 1.2 Subsection ---` |
| Expected | `[Section(1, level, "1. Preparation")]` etc. |
| Actual (before fix) | `[]` |
| Root cause | Stata pattern 1 (`r"^\s*\*{1,}\s*(\d[\d.A-Za-z]*...)"`) handles `* 1. Title` and `*** 1. Title ***` (star decoration only). It does not handle lines with leading dash/equals decoration before the number, as described in the spec's Family 7 (inline decorated). |
| Verdict | **Fixed** |
| Fix | Added pattern 1b to `_STATA_PATTERNS`: `r"^\s*\*+\s*[-=_~]{2,}\s*(\d[\d.A-Za-z]*\.?\s+.+?)\s*[-=_~]{0,}\s*$"`. Only matches numbered content (starts with `\d`) to avoid false positives from attribution lines like `* --- generated by make_table.do ---`. |

## False Negatives Accepted as Limitations

### FN-L01 — Roman numeral section numbering (Stata)

| Input | `* I. Introduction`, `* II. Methods` |
| Expected | Detected as level-1 section |
| Actual | `[]` |
| Root cause | Stata pattern 1 requires `\d` to start the numbered sequence; Roman numerals start with letters. |
| Verdict | **Accept** — Roman numeral sections are rare in Stata research scripts. Adding Roman numeral detection risks false positives (any `* I.` prefix including single-letter variable descriptions). |

---

### FN-L02 — Colon-based headers (Stata)

| Input | `*  Section A: Data preparation`, `* Setup: load data` |
| Expected | Detected as section headers |
| Actual | `[]` |
| Root cause | No Stata pattern handles the `Word: description` format. These look like NOTE or label comments. |
| Verdict | **Accept** — Pattern is ambiguous (e.g., `* NOTE: check this later` should NOT be a section). Colon-based headings are not specified in the pattern spec. |

---

### FN-L03 — Angle-bracket decoration (Python)

| Input | `# >>>>>> Phase 2 >>>>>>` |
| Expected | Detected as section |
| Actual | `[]` |
| Root cause | `>` is not in the recognized decoration character set (`[-=*]`). |
| Verdict | **Accept** — Not in spec. `>` has other meanings in Python (comparison, string formatting); adding it risks false positives. |

### FN-L04 — Tilde decoration (Python)

| Input | `# ~~~ Section ~~~` |

| Actual | `[]` |
| Root cause | `~` is in Stata's decorator set but not in Python's pattern `[-=*]`. |
| Verdict | **Accept** — `~` is a bitwise NOT operator in Python; rare as decoration. |

## False Positives Accepted as Limitations

### FP-L01 — `# TODO: fix this ----` detected as R section

| Input | `# TODO: fix this ----` |
| Expected | `[]` (task annotation, not a section) |
| Actual | `[Section(1, 1, "TODO: fix this")]` |
| Root cause | RStudio trailing-dash pattern captures any comment ending in `----`. TODO annotations ending in `----` are valid RStudio fold markers. |
| Verdict | **Accept** — In R, `# TODO: fix this ----` IS a valid RStudio foldable section. The pattern is correct per RStudio convention. |

---

### FP-L02 — `### FIXME ###` detected as R section

| Input | `### FIXME ###` |
| Expected | `[]` |
| Actual | `[Section(1, 1, "FIXME")]` |
| Root cause | Multi-hash pattern matches `### word ###`. `FIXME` passes all title-validity checks. |
| Verdict | **Accept** — `### FIXME ###` is technically a valid section heading (one could argue it demarcates a fixme region). Filtering specific keywords like FIXME/TODO would be fragile. |
